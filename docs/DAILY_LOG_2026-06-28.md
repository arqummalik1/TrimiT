## 1. Removed PayU, built UPI-intent + Cash payment system

**Removed:** the entire PayU gateway — 22 files deleted across backend, mobile app,
and website.

**New model — dead simple:**
- TrimiT **never holds any customer money**.
- Customers pay the salon directly: either **cash at the counter** or **UPI**
  (Google Pay / PhonePe / Paytm) straight to the salon's own UPI ID.
- For UPI, the app generates a unique reference code (e.g. `TRM-2026-7F3A91`)
  so the salon can match the payment to the booking.
- After paying, the customer lands on a **"Waiting for salon to verify"**
  screen — the app **never says "Payment Successful"**. The salon owner verifies
  with one tap. Only then does the booking confirm.
- The architecture is clean: `PaymentProvider` interface with two concrete
  providers (`UpiIntentManualVerificationProvider`, `CashAtSalonProvider`). If
  we ever add a third method, nothing in the booking engine changes.

**Why we did this:** PayU requires paperwork, a merchant account, and settlement
delays. At v1 with only a few salons, the overhead wasn't worth it. This model
is live, works with every UPI app in India, and lets us launch without a payment
licence.

## 2. Subscription business turned back ON (₹299/month, Razorpay)

The Razorpay subscription system had been switched off. We restored it fully —
identical to the original working commit, byte for byte verified.

**What's live now:**
- Every new salon owner gets a **30-day free trial**, auto-created on signup.
- After trial: **₹299/month**. Razorpay handles the recurring billing.
- **Enforcement is on.** When a trial or subscription lapses:
  - Owner's app is frozen (402 `SUBSCRIPTION_REQUIRED`). They see a "Subscribe"
    screen.
  - Their salon is **greyed out for customers** and shows as unbookable (403
    `SALON_UNAVAILABLE`).
- **Auto-reactivation.** The moment Razorpay's webhook fires (payment success),
  the salon goes live again — no manual step required.
- **Admin escape hatch.** Admin dashboard has a "Grant 30 days" button for
  owners who pay offline or need a manual override.

## 3. UPI required at owner signup

When a salon owner completes their profile, they now must enter a **UPI ID**

## 4. OTP screen flicker — fixed

On Android, the OTP verify screen had a visible flicker when the keyboard
appeared. Root cause: `KeyboardAvoidingView behavior="height"` on Android. Fixed
by switching to `undefined` on Android, `padding` on iOS. Smooth on all devices now.

## 5. Private Admin Dashboard (website only)

Built a founder-only dashboard at `/admin` — not linked anywhere, just type
the URL.

**How to open it:** type the URL → enter your PIN → you're in.

**What it shows:**
- Total customers, owners, salons, bookings (live counters).
- **Owner table:** name, status badge (Trial · N days left / Active / Expired),
  email, joined date, bookings count. One-tap "Grant 30 days" button per owner.
- **Customer table:** name, email, joined date, bookings count.
- **Revenue:** MRR and ARR based on active subscriptions.
- **Website visitors:** page-view chart by day (new `page_views` table, migration
  51). The website tracks a view on every page load via a tiny hook.
- Clean, premium design — readable at a glance.

### Pre-existing broken tests — all fixed (21 suites → 0 failing)

There were ~21 broken mobile test suites left over from a React 19 / Expo 54
upgrade. None were caused by today's work. We went through every one:

**Real bugs found and fixed in app code:**
- Timeout detection: the retry logic only matched `"timeout"` in the message,
  but the error handler writes `"timed out"` → real network timeouts weren't
  being retried. Fixed in `networkRetry.ts` and `userFacingError.ts`.
- Generic 429 rate-limit (e.g. too many booking attempts) was showing the
  auth email-cooldown copy ("check your inbox, wait an hour") — wrong context
  entirely. Fixed in `userFacingError.ts`.
- Navigation helper: a transient dispatch error could surface to the user. Wrapped.
- `normalizeSalon`: `image_url` was set from the raw field, not the normalized
  images array, so if you had both, the thumbnail could differ from the list.
- `Skeleton` component: had a `testID` prop in the type definition but never
  passed it to the DOM — the test runner and any test using `getByTestId` on a
  Skeleton silently couldn't find it.

**Tests updated to match intentional app behaviour:**
- Booking image fallback: when a service has no image, the app shows the
  *category stock image* (same as the salon's service list, for consistency) —
  not the salon hero. Two tests assumed the wrong fallback.
- Map marker: the `pinColor` native prop is intentionally unused (we use a
  custom Ionicons pin to avoid an Android bitmap-crop bug). Tests updated.
- UUIDs in notification tests: Zod v4 validates UUIDs strictly; the tests had
  fake IDs like `1111-1111-1111-...` that aren't valid v4.
- OfflineBanner: jest.mock factory can only reference `mock`-prefixed variables.
- Idempotency: `/bookings` without trailing slash correctly returns false.
- Various: `act()` wrapping, API changes in RNTL 13, dynamic import unsupported in Jest.

**Final count:** 63 suites / 726 tests / 0 failures. TypeScript check clean.

## 7. Map pin color — fixed and made theme-dynamic

The location pins on the salon discovery map were **nearly invisible** — the
unselected state used a near-white color (`#FFF7ED`). Customers couldn't see
where salons were on the map unless they tapped one.

Fixed: pins are now fully theme-driven.
- **Unselected** → `theme.colors.primary` (your brand orange-red, always visible).
- **Selected** → `theme.colors.primaryDark` + the existing pulse animation.

Because the color comes from the theme, if you ever rebrand, every map pin
updates automatically with no extra work.
